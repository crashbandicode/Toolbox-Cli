BNTX writer: byte-identical round-trip on real fixture

Round-trips info_melee __Combined.bntx (1.7 MB, 206 textures) through
read -> write byte-identically.

Major work:

- New full-fidelity BNTX reader that captures every field needed for
  round-trip:
  * BNTX file header (version, alignment_shift, target_address_size,
    flag, first_block_offset, filename_offset)
  * NX header dict_size_field (semantics unclear; preserved verbatim)
  * Container name + complete string pool (an empty sentinel + str_count
    named entries; `read_str_section` parses BntxStr entries with the
    correct alignment formula -- chars+null padded to 2 bytes, NOT 4 as
    jam1garner/bntx writes)
  * Patricia-trie dict captured by string-index (so we don't have to
    derive offsets when round-tripping)
  * Per-texture metadata: every BRTI field including the previously
    undocumented 5 trailing u64s at offsets 0x78..0xA0 (zero, trailing
    block start ptr, trailing block midpoint ptr, zero, zero)
  * BRTD data block as raw bytes
  * Full _RLT relocation table (sections + entries)

- New BNTX writer that recomputes every offset from the parsed state:
  * Memory pool stays at 0x48..0x198, info_ptrs array at 0x198+
  * String pool padded to 8 bytes before _DIC (empirically, real Smash
    BNTX does this so _DIC's 16-byte entries are 8-aligned)
  * BRTI blocks at 0x2A8 stride (0xA0 header + 0x208 trailing). The
    LAST BRTI's `size`/`size2` fields absorb the alignment padding to
    BRTD, so the last block is wider than 0x2A8.
  * BRTD aligned via `align_shift` (= 12 -> 4096) for texture data
  * Relocation table emitted verbatim (works for round-trip; will need
    recomputation when textures are added/removed)

Adds bntx-roundtrip-test internal verb (mirrors the BFLYT one). On the
1.7 MB Smash Ultimate __Combined.bntx fixture: byte-identical (validated
in 1ms after parse).

Bugs found and fixed during development:

- _STR strings use 2-byte alignment for the chars+null block, not 4.
  jam1garner/bntx's reader uses align_after=4 which mismatches the
  on-disk format (their writer is correct because it relies on bnwrite
  semantics that effectively round to 2). Real Smash BNTX shows
  string[1] "__Combined" (10 chars) is 14 bytes total, not 16.

- Total strings in _STR is `str_count + 1` (the empty sentinel is
  written before the strings vector but isn't included in `str_count`).

- BRTI header is 0xA0 bytes; the last 0x30 bytes (offsets 0x70..0xA0)
  contain texture_addr_indirect followed by 5 more u64s (zero +
  trailing-block-start + trailing-block-midpoint + zero + zero). Not in
  the public spec.

- The last BRTI's size field includes alignment padding to BRTD start
  (in our file: 0x1128 vs. 0x2A8 for non-last BRTIs).

- Padding between strings and _DIC pads to 8-byte alignment.
